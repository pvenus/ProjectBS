# Prompt Authoring Guide

## Purpose

Use this guide when creating or updating copy-ready prompts under:

```text
Assets/character_concepts/game_prompts
```

Prompts are user-facing task inputs. They should be short enough to copy into an
agent, but complete enough that the agent can execute the task without reading
unrelated files.

Reference guides, schemas, SO descriptions, enum notes, and long-form policies
must stay under:

```text
Assets/character_concepts/game_prompt_guide
```

## Observed Project Prompt Pattern

Existing prompts generally use this shape:

````text
# {Task Name} Prompt

Short usage note.

## Prompt

```text
작업 폴더 = /Users/pvenus/ProjectBS

{GuideA}.md, {GuideB}.md 기준으로 {task}를 진행해줘.

Input:
- key: {placeholder}

작업:
1. ...

Output:
- ...

검증:
- ...
```
````

Keep this pattern unless the task has a strong reason to differ.

## Prompt File Location

Place copy-ready prompts by domain:

```text
Assets/character_concepts/game_prompts/battle
Assets/character_concepts/game_prompts/character
Assets/character_concepts/game_prompts/skill
Assets/character_concepts/game_prompts/spawner
Assets/character_concepts/game_prompts/stage
Assets/character_concepts/game_prompts/story
```

Do not place prompts under `game_prompt_guide`.

## File Naming

Use one stable task name:

```text
{TaskName}Prompt.md
```

Examples:

```text
EpisodePlanningPrompt.md
BattleFromEpisodePlanPrompt.md
SpawnerCreatePrompt.md
CharacterSOJsonGeneratePrompt.md
```

Use plural `Prompts` only when the file intentionally contains a small prompt
set for one workflow.

## Prompt Scope

Each prompt should execute one independent task or one intentionally bundled
task group.

Good bundles:

- one planning JSON plus its required index updates
- BattleStoryContext JSON plus BattleSO input JSON
- stage JSON plus popup main images
- JSON generation plus validation of the generated JSON

Avoid bundles that cross manual build boundaries:

- Do not bundle BattleSO JSON authoring with Unity asset generation unless the
  prompt is specifically an asset build prompt.
- Do not bundle stage popup generation with BattleSO asset generation unless all
  dependencies already exist and the prompt is explicitly a validation/build
  prompt.
- Do not mix PixelLab image generation with animation download unless the
  prompt is explicitly an end-to-end PixelLab workflow.

## Required Sections

Every prompt should include these sections inside the copy block:

```text
Input:
작업:
Output:
검증:
```

Add these sections when needed:

```text
실패 시 Output:
참조 가이드:
주의:
```

## Input Rules

Inputs should use consistent names across prompts:

```text
actId
chapterId
actGroupId
episodeId
battleId
battleGroup
episodePlanningFile
storyContextFile
episodeCompositionFile
episodeBattleMonsterPoolFile
episodeBattlePlanFile
```

Use placeholder braces for user-provided values:

```text
{act_id}
{chapter_id}
{act_group_id}
{episode_id}
{battle_id}
```

Use real project paths for expected files:

```text
Assets/Doc/StoryPlanning/{act_group_id}/episode.{episode_id}.json
Assets/Resources/battle/{battle_group}/{battle_id}.json
```

Do not hide required dependencies in prose. Put them in `Input`.

## Reference Guide Rules

The prompt should name the minimum guide set required for the task.

Use guide paths when precision matters:

```text
참조 가이드:
- Assets/character_concepts/game_prompt_guide/story/EpisodePlanningCreateGuide.md
- Assets/character_concepts/game_prompt_guide/story/RewardPlanningGuide.md
```

Do not ask the agent to read whole directories. Prefer exact files.

Do not list another prompt as a guide unless the task is explicitly explaining
workflow order. If a prompt depends on another prompt's output, list that output
file as an input instead.

## Task Step Rules

Task steps should be imperative and ordered.

Good:

```text
1. episodePlanningFile에서 battle 항목을 읽는다.
2. 기존 스포너 후보를 검색한다.
3. 후보가 없으면 JSON을 만들지 않고 실패 메시지를 출력한다.
```

Avoid:

```text
1. 적당히 분석한다.
2. 필요하면 여러 문서를 본다.
3. 잘 만든다.
```

When a step can fail, state what to do:

```text
후보가 없으면 JSON을 만들지 말고 실패 메시지를 출력한다.
```

## Output Rules

Outputs should be concrete and checkable:

```text
- 생성/수정한 JSON 경로
- 선택된 spawnerType
- spawnSequenceId
- 검증 결과
```

Avoid output requests like:

```text
- 좋은 결과
- 적절한 요약
```

If the task writes files, every output file path should be named.

## Validation Rules

Validation should test the contract, not restate the task.

Good validation items:

- JSON syntax is valid.
- Referenced input files exist.
- No Unity `.asset` is generated by a JSON-only prompt.
- `rewardId` matches an existing `BattleSO.battleId`.
- `spawnUnitBindings.characterId` resolves to CharacterSO.
- Enum values match code/guide values.
- Generated images follow naming and aspect ratio rules.

## Hard Boundary Rules

State hard boundaries directly in the prompt:

- JSON-only prompts must not create Unity assets.
- Stage JSON must reference battle by `rewardId`; it must not embed BattleSO
  payloads.
- Spawner JSON must not include concrete monster names, `characterId`, or
  `CharacterSO`.
- Episode planning must not decide final spawn counts or final spawner type.
- Battle generation must not invent monsters outside the selected monster pool.
- PixelLab prompts must not substitute another image tool when PixelLab is the
  required source.

## Failure Behavior

For tasks that have dependency gates, include a failure output.

Example:

```text
실패 시 Output:
- 생성하지 않은 파일
- 실패 사유
- 부족한 입력
- 다음에 실행해야 할 프롬프트
```

Do not allow a prompt to create placeholder JSON as a failure artifact unless
the guide explicitly requires a failure artifact.

## Good Prompt Checklist

Before saving a new prompt, confirm:

- It is under `Assets/character_concepts/game_prompts`.
- It has a clear task name.
- It names exact reference guides.
- It has explicit inputs.
- It has ordered task steps.
- It has concrete outputs.
- It has validation criteria.
- It has failure behavior when dependencies can be missing.
- It does not duplicate long guide content.
- It does not ask the agent to read broad folders.
- It does not cross manual build boundaries accidentally.
- It uses current project path conventions.
