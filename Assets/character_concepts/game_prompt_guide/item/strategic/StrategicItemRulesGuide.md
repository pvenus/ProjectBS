# Strategic Item Rules Guide

## 1. Purpose

This guide defines the item-owned data for a strategic item. A strategic item is
a shop/inventory item that spends gauge and invokes one separately generated
EquipmentSkill through `skillId`.

The item JSON does not contain combat profiles, hits, damage, effects, movement,
or visuals for the linked skill.

## 2. Ownership Boundary

| Item owns | Separate skill owns |
|---|---|
| `strategicSkillItemId` | `equipmentId` |
| grade authoring metadata | base profile |
| Korean item name and description | cast and targeting |
| item icon Sprite name | movement and hit behavior |
| gauge cost and reuse policy | damage and effects |
| shop price and tags | skill visuals |
| `skillId` reference | child SO IDs |

Never duplicate a skill field inside the item JSON. The only relationship field
is `skillId`.

## 3. Existing Data Interpretation

The existing strategic shop folder contains 20 legacy `item.json` files whose
skills are embedded. Use their item IDs, prices, costs, and intended gameplay for
comparison only.

- Item IDs use `item.strategic.{slug}`.
- Linked skill IDs use `skill.strategic.{slug}`.
- Gauge costs range from 10 to 100.
- Most prices are 100; one observed price is 150.
- Grades are `Basic` or `Advanced`.
- All current items use `reusable: true`.
- Two descriptions are empty and 19 of 20 tag arrays are empty; these are data
  gaps, not new-authoring precedents.
- The common placeholder-like icon is not a uniqueness rule.

Do not copy the legacy nested `skill` object into a new item JSON.

## 4. Identity and Link Rules

```text
Item ID  = item.strategic.{slug}
Skill ID = skill.strategic.{skill_slug}
```

- Both slugs use lowercase snake case.
- The item slug and skill slug may match, but matching is not required.
- `skillId` must match exactly one generated `EquipmentSkillSO.EquipmentId` under
  Unity `Resources`.
- The linked skill must be generated before the item SO generator is run.
- A missing skill blocks item SO generation; it does not authorize an embedded
  fallback or an arbitrary replacement.
- Duplicate EquipmentSkill IDs are invalid.

## 5. Item Description Rule

`descriptionKo` describes what happens when the item is used, but it must not be
used as an alternative implementation of the skill.

- Every gameplay value in the description must match the linked skill design.
- If the linked skill changes, revalidate the item description.
- Do not invent damage, radius, duration, targets, or effect counts in the item
  workflow.
- Missing linked-skill facts cause `linked_skill_description_mismatch` or
  `missing_linked_skill_design`.

## 6. Gauge and Reuse

The strategic gauge maximum is 100.

```text
0 <= gaugeCost <= 100
reusable = true
```

`gaugeCost=0` requires explicit approval. `reusable=false` is not supported as a
reliable one-use flow by all current call paths, so new items require `true`.

Observed cost bands are review aids, not formulas:

| Cost | Expected tactical weight |
|---:|---|
| 10-25 | narrow utility or small emergency action |
| 30-55 | reliable support, control, or moderate damage |
| 60-80 | strong combined or large-area effect |
| 85-100 | encounter-shaping effect |

The linked skill design is used to review the cost; skill fields are never copied
into the item JSON.

## 7. Price, Grade, Tags, and Icon

- `defaultPrice` is non-negative and comes from economy input.
- `grade` is `Basic` or `Advanced` under the current authoring convention.
- `tags` are stable lowercase classification values.
- `icon` is an exact existing Sprite name.
- Do not reuse the legacy placeholder when the requested icon is missing.

## 8. Generation Order

```text
strategic skill design
→ standalone strategic skill JSON
→ EquipmentSkillSO generation under Resources
→ strategic item JSON containing skillId
→ StrategicSkillItemSO generation
→ runtime lookup by skillId
```

JSON authoring and Unity SO generation are separate operations.

## 9. Duplicate Review

Before creating an item:

1. Check `strategicSkillItemId` against all existing items.
2. Confirm `skillId` resolves to the intended standalone skill.
3. Compare cost, price, grade, tags, and linked behavior with nearby items.
4. Reject an accidental duplicate unless an explicit variant reason exists.

## 10. Validation

- Item JSON contains no `skill` object.
- Item JSON contains exactly one non-empty `skillId` relationship.
- `skillId` resolves to exactly one EquipmentSkillSO in `Resources` before item
  SO generation.
- Item ID, cost, reuse, price, grade, tags, icon, and localization inputs pass.
- Item description matches the separate skill design.
- No skill JSON, child skill SO, effect, visual, image, or prefab is generated by
  the item workflow.
